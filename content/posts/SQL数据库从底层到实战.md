## 前言：图书馆 vs 数据库

你走进一座拥有 100 万本藏书的大型图书馆，想找一本 2018 年出版的、关于"鲨鱼"的书。如果没有管理员、没有检索系统、没有编号规则，你会怎么做？

你只能**一本一本翻**。翻完 100 万本书，你大概需要 3 年。

但如果图书馆有一套**分类编号系统**（就像《杜威十进制分类法》），管理员告诉你："自然科学 → 动物学 → 海洋生物 → 鱼类 → 软骨鱼纲 → 鲨鱼，第 37 排书架第 4 层"，你只需要 5 分钟就能找到那本书。

- 避开新手最常见的 10 个 SQL 坑

## 一、为什么不用 Excel？——数据库存在的理由

### 1.1 Excel 的四个致命缺陷

假设你开了一家网店，用 Excel 管理订单：

| 订单号 | 客户名 | 商品 | 金额 | 状态 |
|--------|--------|------|------|------|
| 001 | 张三 | 鼠标 | 99 | 已发货 |
| 002 | 李四 | 键盘 | 299 | 待支付 |
| 003 | 王五 | 显示器 | 1999 | 已发货 |

刚开始 100 条订单，Excel 挺好用。但到了 10 万条订单时：

| 问题 | Excel 的表现 | 数据库的表现 |
|------|-------------|-------------|
| **并发写入** | 两人同时编辑，后者保存直接覆盖前者 | 通过锁机制，保证你先写入、他后写入，互不覆盖 |
| **数据量** | 超过 100 万行，打开文件就要 30 秒 | 上亿行数据毫秒级响应（有索引的前提下） |
| **数据一致性** | 删除"张三"的订单，但"张三"还在客户表里，出现孤儿数据 | 外键约束自动阻止这种操作 |
| **安全性** | 谁拿到文件谁就能看全部数据 | 可以精细控制：A 只能看自己部门的，B 只能看脱敏后的 |

**一句话总结：Excel 是"个人的数据记录本"，数据库是"多人的数据协作系统"。**

### 1.2 什么是关系型数据库？

**关系型数据库 = 用"表格"来组织数据的数据库。**

"关系"这个词不是指男女关系，而是**数学上的"关系"（Relation）**——说白了就是**二维表**。每一行是一条记录，每一列是一个字段。

主流的关系型数据库：

| 数据库 | 特点 | 典型场景 |
|--------|------|----------|
| MySQL | 开源、免费、社区庞大 | 互联网公司首选，淘宝早期就用它 |
| PostgreSQL | 功能最全、标准兼容最好 | 复杂度高的业务，GIS 地理数据处理 |
| SQLite | 只有一个文件、零配置 | 手机 App（微信聊天记录）、浏览器 |
| Oracle | 商业数据库，贵但稳 | 银行、电信、政府（不差钱的地方） |

本文以 **MySQL** 为例讲解，但 90% 的知识对所有关系型数据库通用。

## 二、数据库的"骨架"：表、行、列、主键

### 2.1 表（Table）= 一个 Excel 工作表

```sql
-- 创建一个"用户"表
CREATE TABLE users (
    id         INT AUTO_INCREMENT PRIMARY KEY,  -- 自增主键
    name       VARCHAR(50)  NOT NULL,           -- 姓名，最长 50 字符，不能为空
    email      VARCHAR(100) UNIQUE,             -- 邮箱，唯一，不能重复
    age        INT          DEFAULT 0,          -- 年龄，默认值 0
    created_at TIMESTAMP    DEFAULT CURRENT_TIMESTAMP  -- 创建时间
);
```

你把这段 SQL 理解为：在 Excel 里新建一个工作表，定义了表头有哪些列，每列的数据类型是什么。

### 2.2 行（Row）= 一条记录

```sql
INSERT INTO users (name, email, age) VALUES ('小明', 'xiaoming@example.com', 25);
```

这就相当于在 Excel 表格里新增一行。

### 2.3 列（Column）= 一个字段

每列有严格的**数据类型**：

| MySQL 类型 | 存什么 | 类比 |
|-----------|--------|------|
| `INT` | 整数（-21亿 ~ 21亿） | 数学考试分数 |
| `BIGINT` | 更大的整数 | 全球人口数量 |
| `VARCHAR(N)` | 可变长字符串，最多 N 个字符 | 姓名、地址 |
| `TEXT` | 长文本（最大 65535 字符） | 文章内容 |
| `DECIMAL(M,D)` | 精确小数（M 位总长，D 位小数） | **金额**（绝不能用 FLOAT） |
| `FLOAT/DOUBLE` | 浮点数（近似值） | 温度、坐标 |
| `DATE` | 日期 | 生日 |
| `DATETIME` | 日期+时间 | 订单创建时间 |
| `BOOLEAN` | 布尔值（实际是 TINYINT） | 是否已读 |
| `BLOB` | 二进制数据 | 图片、文件 |

> ⚠️ **新手最容易犯的错误之一：用 FLOAT 存金额。**
>
> 因为浮点数在计算机里是**近似值**。`0.1 + 0.2` 在计算机里不等于 `0.3`，而是 `0.30000000000000004`。存金额必须用 `DECIMAL`。

### 2.4 主键（Primary Key）= 每行的"身份证号"

主键是**唯一标识一行**的列。就像每个人都有唯一的身份证号，表中每行都有唯一的主键。

```sql
-- id 是主键，每条记录的 id 都不一样
-- 自增（AUTO_INCREMENT）意味着每插入一条新记录，id 自动 +1
SELECT * FROM users WHERE id = 42;  -- 最快找到第 42 号用户
```

**主键的核心作用**：数据库底层用主键来组织数据的物理存储。没有主键的表，数据库会偷偷帮你建一个隐藏的——但这通常不是好实践。**永远显式定义主键。**

## 三、SQL 语句的四大门派

SQL 语句分为四大类，记住它们的英文缩写和对应动作：

| 门派 | 全称 | 中文 | 干什么 | 常用命令 |
|------|------|------|--------|----------|
| **DQL** | Data Query Language | 数据查询 | 查数据 | `SELECT` |
| **DML** | Data Manipulation Language | 数据操作 | 增删改数据 | `INSERT`, `UPDATE`, `DELETE` |
| **DDL** | Data Definition Language | 数据定义 | 定义表结构 | `CREATE`, `ALTER`, `DROP` |
| **DCL** | Data Control Language | 数据控制 | 权限管理 | `GRANT`, `REVOKE` |

> 面试常考题："SQL 语句分为哪几类？" 记住 DQL / DML / DDL / DCL。

其中你 80% 的时间都在写 **DQL**（SELECT 查询），15% 在写 **DML**（增删改），剩下 5% 写 DDL 和 DCL。

## 四、SELECT：从新手到熟练

### 4.1 最基本的查询

```sql
-- 语法结构（这是你一生中写的最多的 SQL 模式）
SELECT 列名1, 列名2
FROM 表名
WHERE 条件
ORDER BY 排序列
LIMIT 条数;
```

**执行顺序**（这是面试高频题）：

```
① FROM     —— 先确定从哪张表查
② WHERE    —— 然后按条件过滤行
③ SELECT   —— 再选出需要的列
④ ORDER BY —— 最后排序
⑤ LIMIT    —— 截取前 N 条
```

> ⚠️ **新手常见误区**：以为 SQL 按书写顺序执行。实际上 `FROM → WHERE → SELECT → ORDER BY → LIMIT`，这个顺序非常关键，后面讲索引时会用到。

### 4.2 WHERE：按条件筛选

```sql
-- 等于
SELECT * FROM users WHERE age = 25;

-- 大于 / 小于
SELECT * FROM users WHERE age > 18 AND age < 60;

-- 模糊匹配（% 代表任意多个字符，_ 代表任意一个字符）
SELECT * FROM users WHERE name LIKE '张%';   -- 姓张的所有人
SELECT * FROM users WHERE name LIKE '张_';   -- 叫"张X"的（两个字）

-- 范围查询
SELECT * FROM users WHERE age BETWEEN 18 AND 30;

-- 集合查询
SELECT * FROM users WHERE city IN ('北京', '上海', '广州');

-- 判空（不能用 = NULL，必须用 IS NULL）
SELECT * FROM users WHERE email IS NULL;       -- 邮箱为空
SELECT * FROM users WHERE email IS NOT NULL;   -- 邮箱不为空
```

> ⚠️ **`= NULL` 永远返回 NULL（即"未知"），不是 TRUE 也不是 FALSE。** 这是 SQL 的三值逻辑：TRUE / FALSE / UNKNOWN。很多新手在这里踩坑。

### 4.3 聚合函数：统计、求和、平均

```sql
-- 统计总人数
SELECT COUNT(*) FROM users;          -- 所有行数
SELECT COUNT(email) FROM users;      -- email 不为 NULL 的行数

-- 年龄统计
SELECT
    MAX(age) AS 最大年龄,            -- AS 起别名
    MIN(age) AS 最小年龄,
    AVG(age) AS 平均年龄,
    SUM(age) AS 年龄总和
FROM users;
```

**COUNT(*) 和 COUNT(列名) 的区别**（面试高频）：
- `COUNT(*)`：统计**所有行**，包括 NULL
- `COUNT(列名)`：统计**该列不为 NULL 的行**
- `COUNT(1)`：跟 `COUNT(*)` 一样，统计所有行

### 4.4 GROUP BY：分组统计

这是日常工作中使用频率最高的查询模式之一：

```sql
-- 每个城市有多少用户？
SELECT city, COUNT(*) AS 人数
FROM users
GROUP BY city;
```

结果：

| city | 人数 |
|------|------|
| 北京 | 1523 |
| 上海 | 2104 |
| 广州 | 987 |

```sql
-- 每个城市、每个年龄段的平均存款
SELECT city, age, AVG(balance) AS 平均存款
FROM users
GROUP BY city, age;
```

### 4.5 HAVING：过滤分组后的结果

**WHERE 和 HAVING 的区别**（面试高频）：

| | WHERE | HAVING |
|---|-------|--------|
| 过滤时机 | **分组前**过滤行 | **分组后**过滤组 |
| 能用聚合函数吗？ | ❌ 不能 | ✅ 能 |

```sql
-- ❌ 错误：WHERE 里不能用聚合函数
SELECT city, COUNT(*) AS cnt FROM users WHERE COUNT(*) > 100 GROUP BY city;

-- ✅ 正确：分组后过滤用 HAVING
SELECT city, COUNT(*) AS cnt
FROM users
GROUP BY city
HAVING COUNT(*) > 100;     -- 只保留人数超过 100 的城市
```

执行顺序完整版：

```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

### 4.6 ORDER BY + LIMIT：排序和分页

```sql
-- 按年龄降序，取前 10 名
SELECT name, age FROM users ORDER BY age DESC LIMIT 10;

-- 分页查询（第 2 页，每页 20 条）
SELECT * FROM users ORDER BY id LIMIT 20 OFFSET 20;
-- LIMIT 20 OFFSET 20 等价于 LIMIT 20, 20
```

> ⚠️ **深分页问题**：`LIMIT 100000, 20` 数据库会先查出 100020 条，再扔掉前 100000 条——非常慢。解决方案见后面的"游标分页"。

## 五、JOIN：多表联查，SQL 的灵魂

### 5.1 为什么需要多张表？

假设你只有一张"万能表"：

| 订单号 | 客户名 | 客户电话 | 商品名 | 商品价格 | 商品库存 |
|--------|--------|----------|--------|----------|----------|
| 001 | 张三 | 138... | 鼠标 | 99 | 500 |
| 002 | 张三 | 138... | 键盘 | 299 | 200 |
| 003 | 李四 | 139... | 鼠标 | 99 | 500 |

问题很明显：
- 张三的信息重复存了两遍（冗余）
- 如果张三换手机号，要改 N 个地方
- 如果鼠标价格变了，也要改 N 个地方

**解决方案：拆表。**

```
users 表：            orders 表：              products 表：
┌────┬──────┬─────┐  ┌────┬──────┬────────┬─┐ ┌────┬──────┬──────┐
│ id │ name │ tel │  │ id │user_id│prod_id │ │ │ id │ name │price │
├────┼──────┼─────┤  ├────┼──────┼────────┼─┤ ├────┼──────┼──────┤
│ 1  │ 张三 │138..│  │ 1  │  1   │   1    │ │ │ 1  │ 鼠标 │  99  │
│ 2  │ 李四 │139..│  │ 2  │  1   │   2    │ │ │ 2  │ 键盘 │ 299  │
└────┴──────┴─────┘  │ 3  │  2   │   1    │ │ └────┴──────┴──────┘
                     └────┴──────┴────────┴─┘
```

现在张三改手机号只需要改 `users` 表一行。但查询时要把三张表"拼起来"看——这就是 **JOIN**。

### 5.2 JOIN 的类型（用维恩图理解）

```
INNER JOIN:        LEFT JOIN:          RIGHT JOIN:         FULL OUTER JOIN:
┌─────┐           ┌─────────┐         ┌─────────┐         ┌───────────────┐
│A│═══│B│         │A │═══│ B│         │  A ═══│B │         │  A │═══│  B  │
│ │ B │ │         │  │ B │  │         │  │  B │  │         │  │ B.  │  │
└──┘───┘──┘       └──┘────┘──┘       └──┘─────┘──┘       └──┘────────┘──┘
 A∩B 交集          A全部 + B匹配        A匹配 + B全部        A和B全部
```

```sql
-- 内连接：只返回两表都匹配的行
SELECT users.name, orders.id AS order_id
FROM users
INNER JOIN orders ON users.id = orders.user_id;
-- 结果：只有下过单的用户

-- 左连接：返回左表全部行 + 右表匹配的行（右表没匹配的填 NULL）
SELECT users.name, orders.id AS order_id
FROM users
LEFT JOIN orders ON users.id = orders.user_id;
-- 结果：所有用户，没下过单的 order_id 显示 NULL

-- 多表连接
SELECT u.name, o.id AS order_id, p.name AS product_name
FROM users u
JOIN orders o    ON u.id = o.user_id
JOIN products p  ON o.product_id = p.id;
-- 表别名：users u 就是给 users 表起别名 u，简化写法
```

### 5.3 JOIN 的底层原理：数据库怎么把两张表拼起来？

面试官问："JOIN 是怎么执行的？" 你需要知道三种算法：

#### 算法一：Nested Loop Join（嵌套循环连接）——最基础

```
for 左表的每一行:
    for 右表的每一行:
        if 匹配条件:
            输出这一对
```

如果左表 1000 行，右表 10000 行，最坏情况需要比较 1000 × 10000 = 1000 万次。**没有索引时，这就是 JOIN 慢的根源。**

#### 算法二：Index Nested Loop Join（索引嵌套循环连接）——有索引时用

```
for 左表的每一行:
    用左表的值去右表的索引里查（B+ 树查找，O(log n)）
```

右表有索引时，每行匹配只需要 `O(log 10000) ≈ 14` 次比较，总共约 14000 次——快了 700 倍。

#### 算法三：Hash Join（哈希连接）——MySQL 8.0 新增

```
1. 把较小的表建成哈希表（key = 连接列的值，value = 行数据）
2. 遍历大表的每一行，用连接列的值去哈希表里查（O(1)）
```

**这就是为什么 JOIN 的 ON 条件列一定要建索引。**

## 六、索引：数据库的"目录"

### 6.1 没有索引的世界有多惨

假设 `users` 表有 100 万行数据，没有索引：

```sql
SELECT * FROM users WHERE name = '张三丰';
```

数据库只能**全表扫描（Full Table Scan）**——从第 1 行开始，一行一行比对 `name`，直到找到（或找完 100 万行）。平均需要扫描 **50 万行**。

有了索引后，就像翻书查目录：先在索引里找到"张三丰"在哪一页，直接翻过去。只需要 **3~4 次磁盘 I/O**。

**快了 10 万倍。**

### 6.2 索引到底是什么？

**一句话：索引是一种"排序的数据结构"，让你能快速找到数据，而不需要遍历全表。**

它本质上是**用空间换时间**——多存一份排好序的数据，来换取查询速度。

### 6.3 B+ 树：MySQL 索引的底层数据结构

这是面试必问的内容，也是理解索引优化的关键。

#### 为什么不用二叉搜索树？

二叉搜索树（BST）在极端情况下会退化成链表：

```
平衡的 BST：          退化的 BST（插入 1,2,3,4,5）：
      3                  1
     / \                  \
    2   4                  2
   /     \                  \
  1       5                  3
                               \
                                4
                                 \
                                  5
  查找：4 步              查找：5 步（跟遍历没区别）
```

而且 BST 每个节点只有两个子节点，树会很高。树越高，查询需要访问的节点越多 → 磁盘 I/O 越多 → 越慢。

#### 为什么选 B+ 树？

B+ 树是 B 树的变种，专门为**磁盘存储**优化：

```
                     [30 | 70]                 ← 根节点（存在内存里）
                    /    |    \
           [10|20]     [40|50|60]    [80|90]   ← 中间节点
          /   |   \    /  |  |  \   /   |   \
        [1] [10] [20] [30] [40] [50] [60] [70] [80] [90] ...  ← 叶子节点（真正存数据）
         |    |    |    |    |    |    |    |    |    |
        数据  数据  数据  数据  数据  数据  数据  数据  数据  数据
        ↑                                            ↑
        所有叶子节点通过链表连接（支持范围查询）
```

**B+ 树的特点**：

| 特点 | 含义 | 为什么重要 |
|------|------|-----------|
| **多叉** | 每个节点可以有成百上千个子节点 | 树很"矮胖"，通常只有 3~4 层 |
| **所有数据都在叶子节点** | 非叶子节点只存索引值，不存数据 | 非叶子节点可以存更多索引 → 树更矮 |
| **叶子节点有序 + 链表** | 叶子节点按索引值排序，用双向链表连接 | 范围查询 `BETWEEN 10 AND 100` 只需定位起点然后顺着链表走 |
| **节点大小 = 磁盘页大小** | 一个节点刚好填满一个磁盘页（16KB） | 一次磁盘 I/O 读一个完整节点 |

**为什么 3~4 层的 B+ 树能存千万级数据？**

假设每个索引项 14 字节（8 字节 BIGINT 主键 + 6 字节指针），一个 16KB 的节点大约能存 1170 个索引项。

```
根节点：1170 个子节点
第 2 层：1170 × 1170 = 136 万个子节点（或直接是叶子）
第 3 层（叶子）：136 万 × 16（每条记录约 1KB）= 2176 万条记录
```

3 层 B+ 树就能存 2000 万条记录，找任何一条只需要 **3 次磁盘 I/O**。

### 6.4 创建索引

```sql
-- 单列索引
CREATE INDEX idx_name ON users(name);

-- 复合索引（多列组合成一个索引）
CREATE INDEX idx_city_age ON users(city, age);

-- 唯一索引（索引列的值不能重复）
CREATE UNIQUE INDEX idx_email ON users(email);

-- 建表时直接指定
CREATE TABLE users (
    id INT PRIMARY KEY,                  -- 主键自动创建索引
    email VARCHAR(100) UNIQUE,            -- UNIQUE 自动创建索引
    name VARCHAR(50),
    INDEX idx_name (name)                 -- 手动创建普通索引
);
```

### 6.5 最左前缀原则：复合索引最重要的规则

假设你建了一个复合索引 `(city, age, name)`：

```sql
-- ✅ 走索引（匹配最左列 city）
SELECT * FROM users WHERE city = '北京';

-- ✅ 走索引（匹配最左两列 city + age）
SELECT * FROM users WHERE city = '北京' AND age = 25;

-- ✅ 走索引（匹配全部三列）
SELECT * FROM users WHERE city = '北京' AND age = 25 AND name LIKE '张%';

-- ❌ 不走索引（跳过了 city，最左列没匹配）
SELECT * FROM users WHERE age = 25;

-- ❌ 不走索引（跳过了 city 和 age）
SELECT * FROM users WHERE name = '张三';
```

**类比**：复合索引就像**多层目录**——先按省分类，再按市分类，再按区分类。你想查"朝阳区"，如果不告诉目录系统是哪个省哪个市，目录系统也无能为力。

### 6.6 覆盖索引：查询不用回表

```sql
-- 假设有索引 (city, age, name)

-- ❌ 需要回表：索引里没有 email，查到后还要回主键索引取 email
SELECT * FROM users WHERE city = '北京' AND age = 25;

-- ✅ 覆盖索引：SELECT 的列全在索引里，不用回表
SELECT city, age, name FROM users WHERE city = '北京' AND age = 25;
```

**回表**：通过辅助索引找到主键，再拿着主键去主键索引里查完整数据——多一次磁盘 I/O。覆盖索引避免了这次 I/O，是最有效的 SQL 优化手段之一。

> 使用 `EXPLAIN` 查看执行计划时，`Extra` 列出现 `Using index` 就表示使用了覆盖索引。

### 6.7 EXPLAIN：看数据库怎么执行你的 SQL

```sql
EXPLAIN SELECT * FROM users WHERE name = '张三';
```

结果的关键列：

| 列名 | 含义 | 关注点 |
|------|------|--------|
| `type` | 访问类型 | **从好到坏**：`const` > `eq_ref` > `ref` > `range` > `index` > `ALL` |
| `key` | 实际使用的索引 | NULL = 没用索引，全表扫描 |
| `rows` | 预估扫描行数 | 越小越好 |
| `Extra` | 额外信息 | `Using index`（覆盖索引，好）、`Using filesort`（文件排序，差）、`Using temporary`（临时表，差） |

> ⚠️ 看到 `type: ALL`（全表扫描）且 `rows` 很大 → 这条 SQL 有问题，需要加索引或改写。

### 6.8 索引不是越多越好

每插入一条数据，**所有索引都要更新**。索引多 → 写入慢 → 磁盘空间多。

**建索引的原则**：
- WHERE、JOIN、ORDER BY 后面的列考虑建索引
- 区分度低的列（如性别只有男/女）建索引效果很差——数据库宁愿全表扫描
- 频繁更新的列慎重建索引
- 复合索引的列顺序：**区分度高的放前面**

## 七、事务：银行转账为什么不会丢钱？

### 7.1 什么是事务？

**事务是一组 SQL 操作，要么全部成功，要么全部失败。**

最经典的例子——A 给 B 转 100 元：

```sql
-- 步骤 1：A 账户减 100
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- 💥 假设这里数据库崩溃了！

-- 步骤 2：B 账户加 100
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
```

如果没有事务，A 的钱扣了 100，但 B 没收到——**100 块钱凭空消失了。**

有事务时：

```sql
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;  -- 提交：两个修改一起生效
-- 或者 ROLLBACK; -- 回滚：两个修改一起撤销
```

### 7.2 ACID：事务的四大特性

| 特性 | 全称 | 一句话解释 | 类比 |
|------|------|-----------|------|
| **A** | Atomicity 原子性 | 要么全做，要么全不做 | 结婚登记——要么两人都签字，要么都不签 |
| **C** | Consistency 一致性 | 事务前后，数据都符合规则 | 总余额不变——A 少了 100，B 必须多 100 |
| **I** | Isolation 隔离性 | 事务之间互不干扰 | 你在 ATM 取钱时，别人不能同时取同一笔 |
| **D** | Durability 持久性 | 一旦提交，数据就永久保存 | 结婚证领了就是领了，停电也不会变回单身 |

### 7.3 隔离级别：从"完全不隔离"到"完全串行"

隔离性越强，数据越安全，但并发性能越差。数据库给你 4 个档位：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|----------|------|-----------|------|------|
| READ UNCOMMITTED（读未提交） | ✅ | ✅ | ✅ | 最快 |
| READ COMMITTED（读已提交） | ❌ | ✅ | ✅ | 较快 |
| REPEATABLE READ（可重复读） | ❌ | ❌ | ✅（MySQL 中部分解决） | 中等 |
| SERIALIZABLE（串行化） | ❌ | ❌ | ❌ | 最慢 |

**三个并发问题通俗解释**：

| 问题 | 通俗解释 | 举例 |
|------|----------|------|
| **脏读** | 读到了别人**还没提交**的数据 | A 转账还没确认，B 就看到余额变了；后来 A 回滚了，B 看到的是假余额 |
| **不可重复读** | 同一个事务内，两次读到的数据**不一样** | 第一次查余额 1000，中间别人转走 500 并提交，第二次查变成 500 |
| **幻读** | 同一个事务内，两次查询的**行数不一样** | 第一次查有 10 个订单，中间别人插入了 1 条并提交，第二次查变成 11 条 |

MySQL 默认隔离级别是 **REPEATABLE READ**。查看和设置：

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### 7.4 MVCC：MySQL 怎么实现"读写不冲突"？

**MVCC（Multi-Version Concurrency Control，多版本并发控制）**是数据库最精巧的设计之一。

**核心思想**：不直接在原数据上修改，而是保留数据的**多个版本**。每个事务看到的是数据的**一个快照**。

类比：
- 你在 10:00 开始看一份报告，即使别人在 10:05 修改了它，你看到的仍然是 10:00 的版本
- 你修改完提交时，数据库检查"有没有别人在你看完之后改了同一行？"
- 如果有冲突 → 你的修改冲突，需要重试
- 如果没有 → 你的修改成功

MVCC 的具体实现依赖两个隐藏列：

```
每一行数据背后，MySQL 实际存了：
┌────┬──────┬──────┬───────────┬───────────┬──────────────┐
│ id │ name │ age  │ trx_id    │ roll_ptr  │ 其他...       │
├────┼──────┼──────┼───────────┼───────────┼──────────────┤
│  1 │ 张三 │  25  │  1001     │ → 指向上一个版本 │             │
└────┴──────┴──────┴───────────┴───────────┴──────────────┘

trx_id：最后一次修改这行的事务 ID
roll_ptr：指向这行在 undo log 中的上一个版本（回滚段指针）
```

通过 `trx_id` 和 `roll_ptr`，MySQL 可以"看到"这行数据的任意历史版本，从而实现：
- **读不阻塞写**：读操作直接读快照，不等写操作释放锁
- **写不阻塞读**：写操作在最新版本上修改，不阻塞读快照

### 7.5 锁：事务隔离的底层实现

MySQL InnoDB 引擎使用多种锁来实现隔离：

| 锁类型 | 粒度 | 特点 |
|--------|------|------|
| **行锁（Record Lock）** | 一行 | 锁定一行数据，并发性最好 |
| **间隙锁（Gap Lock）** | 行之间的间隙 | 锁定一个范围，防止别人在这个范围插入数据（解决幻读） |
| **临键锁（Next-Key Lock）** | 行 + 前面的间隙 | 行锁 + 间隙锁的组合，InnoDB 默认的行级锁 |
| **表锁（Table Lock）** | 整张表 | 简单粗暴，并发性差 |

```sql
-- 查看当前锁状态
SELECT * FROM performance_schema.data_locks;

-- 死锁示例（两个事务互相等待对方释放锁）
-- 事务 A：UPDATE users SET age=26 WHERE id=1;  -- 锁住 id=1
-- 事务 B：UPDATE users SET age=27 WHERE id=2;  -- 锁住 id=2
-- 事务 A：UPDATE users SET age=28 WHERE id=2;  -- 等 B 释放锁
-- 事务 B：UPDATE users SET age=29 WHERE id=1;  -- 等 A 释放锁 → 死锁！
```

MySQL 会自动检测死锁并回滚其中一个事务，所以你的代码里需要处理"事务被回滚"的情况（重试逻辑）。

## 八、一条 SQL 的完整旅程

当你输入 `SELECT * FROM users WHERE id = 42;` 按下回车，数据库内部发生了什么？

```
┌─────────────────────────────────────────────────────────────────┐
│  1. 客户端发送 SQL                                                │
│     mysql> SELECT * FROM users WHERE id = 42;                    │
└───────────────────────┬─────────────────────────────────────────┘
                        │ TCP 连接
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  2. 连接器：验证用户名密码，检查权限                                  │
│     "你是谁？你有权限查 users 表吗？"                               │
└───────────────────────┬─────────────────────────────────────────┘
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  3. 查询缓存（MySQL 8.0 已废弃）                                    │
│     "这条 SQL 之前查过吗？结果是不是一样的？"                           │
│     （缓存命中率极低，因为表一更新缓存就失效，所以被干掉了）               │
└───────────────────────┬─────────────────────────────────────────┘
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  4. 分析器（Parser）：词法分析 + 语法分析                             │
│     "SELECT 是关键字，* 是所有列，FROM 后面是表名，WHERE 后面是条件"    │
│     如果 SQL 写错了，这一步就报错                                    │
└───────────────────────┬─────────────────────────────────────────┘
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  5. 优化器（Optimizer）：决定"怎么查最快"                            │
│     "WHERE id = 42 → id 是主键 → 走主键索引 → 1 次 B+ 树查找"       │
│     优化器会评估多个方案，选成本最低的那个                              │
└───────────────────────┬─────────────────────────────────────────┘
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  6. 执行器（Executor）：真正干活                                    │
│     调用存储引擎接口：                                              │
│     ha_innobase::index_read(id=42)                               │
│         → 从根节点开始，根据主键树找到第 42 行所在的叶子页             │
│             → 读磁盘（如果不在 Buffer Pool）                        │
│                 → 返回这一行的全部列                                 │
└───────────────────────┬─────────────────────────────────────────┘
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  7. 返回结果给客户端                                               │
│     +----+--------+------+                                       │
│     | id | name   | age  |                                       │
│     +----+--------+------+                                       │
│     | 42 | 张三丰 |  108 |                                       │
│     +----+--------+------+                                       │
└─────────────────────────────────────────────────────────────────┘
```

### InnoDB 存储引擎的关键组件

| 组件 | 是什么 | 作用 |
|------|--------|------|
| **Buffer Pool**（缓冲池） | 一块大内存 | 缓存热点数据页，减少磁盘 I/O。绝大多数查询直接命中内存 |
| **Change Buffer**（写缓冲） | 内存中的暂存区 | 对非唯一二级索引的修改先写在这里，后面异步合并到磁盘 |
| **Redo Log**（重做日志） | 磁盘上的环形日志 | 记录"数据被改成了什么"，崩溃恢复时用。**保证持久性** |
| **Undo Log**（回滚日志） | 记录"数据以前是什么" | 回滚事务 + MVCC 读历史版本。**保证原子性** |
| **Doublewrite Buffer**（双写缓冲） | 磁盘上的一块区域 | 防止"页部分写坏"——先写到这里，再写到真正的数据文件 |
| **Adaptive Hash Index**（自适应哈希） | 自动在内存中建的哈希索引 | 频繁访问的 B+ 树路径自动建哈希索引，进一步加速 |

> **Buffer Pool 是性能核心**：生产环境的 MySQL，Buffer Pool 通常配置为物理内存的 50%~80%。绝大多数读操作不碰磁盘，直接在内存完成。

## 九、SQL 优化：从"跑得动"到"跑得快"

### 9.1 先学会看慢查询

```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过 1 秒就算慢查询

-- 查看慢查询日志位置
SHOW VARIABLES LIKE 'slow_query_log_file';
```

### 9.2 十大优化法则

| 法则 | 怎么做 | 为什么 |
|------|--------|--------|
| ❶ **建对索引** | WHERE、JOIN、ORDER BY 的列建索引 | 避免全表扫描 |
| ❷ **避免 SELECT \*** | 只查需要的列 | 减少网络传输 + 可能用到覆盖索引 |
| ❸ **最左前缀** | 复合索引按最左列查询 | 否则索引不生效 |
| ❹ **避免在索引列上做运算** | `WHERE age+1=26` → `WHERE age=25` | 索引列参与运算会导致索引失效 |
| ❺ **隐式类型转换** | `WHERE phone=13800138000` → `WHERE phone='13800138000'` | phone 是 VARCHAR，跟整数比较会触发类型转换 → 索引失效 |
| ❻ **LIKE 以 % 开头** | `LIKE '%张'` 索引失效，`LIKE '张%'` 走索引 | B+ 树只能前缀匹配 |
| ❼ **大分页优化** | `LIMIT 100000,20` → 用"游标分页" | 避免数据库扔掉 10 万行 |
| ❽ **批量操作** | 100 条 INSERT 合并成一个事务 | 减少刷盘次数 |
| ❾ **EXPLAIN 检查** | 上线前每条 SQL 跑一遍 EXPLAIN | 提前发现全表扫描 |
| ❿ **避免大事务** | 一个事务别包含几十个 UPDATE | 长事务锁住资源，阻塞其他人 |

### 9.3 大分页优化详解

```sql
-- ❌ 传统分页：OFFSET 越大越慢
SELECT * FROM users ORDER BY id LIMIT 20 OFFSET 100000;
-- 实际执行：查出 100020 行 → 扔掉前 100000 行 → 返回 20 行

-- ✅ 游标分页（Keyset Pagination）：记录上一页最后一条的 id
SELECT * FROM users WHERE id > 100000 ORDER BY id LIMIT 20;
-- 实际执行：从 id=100000 开始，走索引直接读 20 行
```

## 十、SQL 注入：每个新手都必须知道的安全底线

### 10.1 什么是 SQL 注入

假设你的登录逻辑是这样写的（**错误示范**）：

```python
# ❌ 致命错误：拼接用户输入到 SQL
username = request.form['username']  # 用户输入
password = request.form['password']
sql = f"SELECT * FROM users WHERE name='{username}' AND password='{password}'"
```

正常用户输入 `username=admin, password=123456`：

```sql
SELECT * FROM users WHERE name='admin' AND password='123456'
```

但如果攻击者输入 `username=admin' -- `（注意 `--` 是 SQL 注释符）：

```sql
SELECT * FROM users WHERE name='admin' -- ' AND password='whatever'
-- `-- ` 后面的全被注释掉了！不需要密码就能登录！
```

### 10.2 防范：参数化查询

```python
# ✅ 正确：参数化查询（也叫预编译语句）
cursor.execute("SELECT * FROM users WHERE name=? AND password=?", (username, password))
# 数据库把 ? 位置的值当作"纯数据"，永远不会当作 SQL 代码执行
```

**永远不要拼接用户输入到 SQL 语句里。永远用参数化查询。** 这句话请默念三遍。

## 十一、新手常见的 10 个 SQL 错误

| # | 错误写法 | 问题 | 正确写法 |
|---|----------|------|----------|
| 1 | `WHERE name = NULL` | `= NULL` 永远返回 UNKNOWN | `WHERE name IS NULL` |
| 2 | `SELECT *` 满天飞 | 传输多余数据、无法利用覆盖索引 | 只查需要的列 |
| 3 | 用 FLOAT 存金额 | 浮点数有精度问题 | 用 `DECIMAL(10,2)` |
| 4 | 没建索引就上线 | 数据量一大，查询就慢 | 上线前分析查询模式，预先建索引 |
| 5 | 复合索引不遵循最左前缀 | 索引不生效 | 按最左列组织查询 |
| 6 | `LIKE '%搜索词%'` | 前后都有 `%`，索引失效 | 尽量用 `LIKE '搜索词%'`；全模糊用全文索引 |
| 7 | 拼接用户输入到 SQL | SQL 注入漏洞 | 用参数化查询 |
| 8 | 在索引列上做函数运算 | `WHERE YEAR(created_at)=2024` 索引失效 | `WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'` |
| 9 | 不控制事务大小 | 一个事务锁住全表几秒 | 事务尽量小，尽快提交 |
| 10 | 不写注释，表名/列名随意 | `t1`, `a`, `b` 三个月后自己都看不懂 | 用有意义的英文命名，加 COMMENT |

## 十二、总结：从新手到进阶的学习路径

### 本文核心回顾

```
数据库是什么？        → 多用户协作的数据管理系统，不是大号 Excel
表、行、列、主键       → 数据的基本组织方式
SQL 四大类            → DQL（查）DML（增删改）DDL（定义表）DCL（权限）
SELECT 执行顺序       → FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
JOIN                 → 把多张表按条件拼起来，ON 列要建索引
索引                 → B+ 树，空间换时间，最左前缀原则
事务                 → ACID，MVCC 实现读不阻塞写
SQL 执行链路         → 连接器 → 分析器 → 优化器 → 执行器 → 存储引擎
```

### 进阶学习路线

1. **把本文的 SQL 都敲一遍** — 在你的电脑上装个 MySQL，建表、插数据、查、改、删、建索引
2. **学会看 EXPLAIN 输出** — 这是你调优 SQL 的眼睛
3. **阅读《高性能 MySQL》** — 业界圣经，尤其重点看第 4 章（Schema 优化）和第 5 章（查询优化）
4. **了解 InnoDB 源码** — 推荐先看 Buffer Pool 和 B+ 树的实现，这是最核心的两个模块
5. **实践** — 在你自己的项目里留意慢查询，尝试优化

---

> **数据库不是什么高深莫测的东西。它只是一个帮你管数据的工具，内部原理说到底就是：把数据组织好（B+ 树），读写协调好（MVCC + 锁），崩溃了能恢复（Redo Log）。** 理解了这三件事，你就真正理解了数据库。
