## 前言：一个餐厅的故事

你开了一家餐厅，这家餐厅有一种特殊的经营模式：

- 顾客进门后，坐进座位就开始**点菜**。一个顾客可能点 1 道菜，也可能点 100 道菜。
- 厨房里有**厨师**，厨师负责做菜。
- 每个厨师面前有一个**灶台**，灶台上能放好几口锅。厨师一次只能盯着一口锅，但可以随时换另一口锅翻炒。

问题来了：

- 顾客（任务）数量从几十到几万不等，远超厨师数量
- 厨师（线程）如果每来一个顾客就新雇一个，工资成本会爆炸
- 如果一口锅在等食材（等待 IO），厨师不应该傻站着——他应该去炒另一口锅的菜

**这就是 Go 的 GMP 调度模型要解决的问题。**

| 餐厅概念 | GMP 概念 | 说明 |
|---------|---------|------|
| 顾客点的菜 | **G（Goroutine）** | 一个需要执行的任务 |
| 厨师 | **M（Machine）** | 操作系统线程 |
| 灶台 | **P（Processor）** | 逻辑处理器，持有可运行的 G 队列 |

> 本文假定你是一个 Go 新手，对并发只有模糊的概念。我会用大量比喻和源码片段，带你从零看懂 GMP。读完你不仅能应付面试，还会真正理解 Go 为什么能"百万并发"。

读完本文你将能够：

- 用大白话解释 GMP 三个字母分别是什么、为什么需要它们
- 理解 Go 调度器从 GM 到 GMP 的演进历程
- 画出 GMP 调度的完整流程图
- 看懂 `runtime` 包中 GMP 的核心源码结构
- 回答 GMP 相关的经典面试题

## 一、前置知识：别急着看 GMP，先搞懂并发

### 1.1 进程、线程、协程——三个层次的关系

很多人搞不清这三者，我用一个公司比喻讲清楚：

| 层次 | 比喻 | 操作系统视角 |
|------|------|-------------|
| **进程** | 一家公司 | 独立的内存空间、文件描述符。进程间通信需要 IPC（管道、共享内存等）。 |
| **线程** | 公司里的一个员工 | 共享进程的内存空间。线程切换由操作系统内核完成，成本较高（约 1~10μs）。 |
| **协程** | 员工手上的多个任务 | 用户态"轻量级线程"，切换由用户代码控制，成本极低（约几十 ns）。 |

关键数字记住就行：

```
进程切换：慢得离谱（切换页表、刷新 TLB）
线程切换：~1-10 微秒（内核态切换，保存/恢复寄存器）
协程切换：~几十纳秒（用户态，只保存/恢复少量寄存器）
```

Go 的 goroutine 就是一种**协程**，但它不是普通的协程——Go 的调度器会帮你把 goroutine 自动分配到线程上执行，你不用操心。

### 1.2 为什么要有调度器？

一句话：**调度器的本质任务，就是把 M 个任务高效地分配到 N 个执行单元上。**

操作系统的线程调度器做的是：把几千个线程分配到几个 CPU 核心上。Go 的调度器做的是：把几十万个 goroutine 分配到几个操作系统线程上。

为什么要再加一层？因为操作系统线程调度器有这些问题：

1. **线程切换太慢**：每次线程切换都要陷入内核态
2. **线程太贵**：每个线程初始栈至少 1MB，创建 10000 个线程就要 10GB 内存
3. **调度不够灵活**：内核调度器不知道你 goroutine 之间的关系

Go 的答案：**在用户态再做一个调度器，把大量 goroutine 映射到少量线程。**

## 二、GM 模型：Go 调度器的 1.0 版本

在 GMP 之前，Go 有一个更简单的版本——GM 模型。

### 2.1 GM 模型长什么样

```
┌─────────────────────────────────────────┐
│              全局 Goroutine 队列          │
│   [G1] [G2] [G3] [G4] [G5] ... [Gn]    │
└───────┬─────────────────────────────────┘
        │  所有 M 都从这里取 G
        ▼
┌───────────────┐  ┌───────────────┐  ┌───────────────┐
│      M1       │  │      M2       │  │      M3       │
│  (OS Thread)  │  │  (OS Thread)  │  │  (OS Thread)  │
└───────────────┘  └───────────────┘  └───────────────┘
```

**全局队列 + 多线程竞争取任务**，看起来简单直接，但问题很大。

### 2.2 GM 模型的三大硬伤

**硬伤一：全局锁竞争**

所有 M 都去同一个全局队列抢 G，这个全局队列必须加锁保护。当 goroutine 数量很大时，锁竞争会成为瓶颈——M 们大量时间花在抢锁上，而不是执行 G。

```go
// 伪代码：GM 模型的问题
var globalQueue []*G
var mu sync.Mutex

func (m *M) schedule() {
    for {
        mu.Lock()
        if len(globalQueue) > 0 {
            g := globalQueue[0]
            globalQueue = globalQueue[1:]
            mu.Unlock()
            g.run()  // 执行 goroutine
        } else {
            mu.Unlock()
        }
    }
}
```

每个 M 都要抢这把锁，就像 100 个厨师排队在一个窗口取菜单——窗口前挤满了人，真正做菜的时间反而很少。

**硬伤二：内存局部性差**

M1 刚执行完 G1，再取 G2 时，G2 的内存数据大概率不在 CPU 缓存里。因为 G1 和 G2 之间没有任何关系，谁抢到谁执行。这导致 CPU 缓存命中率低，性能差。

**硬伤三：M 数量不可控**

当某个 M 执行 G 时发生系统调用阻塞（比如读文件），这个 M 就被操作系统挂起了。此时全局队列里还有 G 在等待，系统会创建新的 M 来执行——但创建多少 M 没有上限，可能导致线程数量爆炸。

### 2.3 从 GM 到 GMP：引入了什么？

Go 1.1 引入了 **P（Processor）**，核心改动一句话：

> **给每个 M 配一个私有的 G 队列，让 M 大部分时间不用去全局队列抢。**

这就是 GMP 模型的由来：

```
       全局 G 队列 (偶尔访问)
    ┌─────────────────────────┐
    │  G10  G11  G12  ...     │
    └─────────────────────────┘
              ▲
              │ 偷取 (work stealing)
    ┌─────────┼─────────┐
    │         │         │
    ▼         ▼         ▼
┌───────┐ ┌───────┐ ┌───────┐
│  P0   │ │  P1   │ │  P2   │  每个 P 有自己的本地 G 队列
│ [G1]  │ │ [G4]  │ │ [G7]  │
│ [G2]  │ │ [G5]  │ │ [G8]  │
│ [G3]  │ │ [G6]  │ │ [G9]  │
└───┬───┘ └───┬───┘ └───┬───┘
    │         │         │
    ▼         ▼         ▼
┌───────┐ ┌───────┐ ┌───────┐
│  M1   │ │  M2   │ │  M3   │
└───────┘ └───────┘ └───────┘
```

**P 的数量默认等于 CPU 核心数**（通过 `GOMAXPROCS` 设置）。为什么？因为一个 CPU 核心同一时刻只能执行一条指令，设置更多 P 只会增加调度开销，不会让程序更快。

## 三、G、M、P 逐个拆解：它们到底是什么

### 3.1 G（Goroutine）——被调度的基本单位

G 代表一个 goroutine，它是 Go 调度器的**最小调度单位**。你可以把 G 理解为"一个待执行的任务"。

**G 里面存了什么？（源码级别）**

```go
// src/runtime/runtime2.go
type g struct {
    // 栈相关信息
    stack       stack   // 当前 goroutine 的栈内存范围 [lo, hi)
    stackguard0 uintptr // 栈溢出检测标记
    
    // 调度信息
    m         *m       // 当前在哪个 M 上执行（没有执行则为 nil）
    sched     gobuf    // 调度上下文：被换下 CPU 时，sp/pc/等寄存器保存在这里
    atomicstatus uint32 // G 的状态（运行中/可运行/等待中/等等）
    goid      int64    // goroutine 的唯一 ID
    
    // 指针与链接
    schedlink guintptr // 链表指针。P 的本地队列本质是一个 G 的链表，通过这个字段串起来
    
    // 抢占相关
    preempt       bool // 是否应该被抢占
    preemptStop   bool // 抢占时是否应该挂起
    
    // 等等...
}
```

**G 的生命周期状态机：**

```
                    go func() 创建
                         │
                         ▼
    ┌─────────────────────────────────────┐
    │            _Gidle（刚创建）           │
    └─────────────────────────────────────┘
                         │  加入 P 的本地队列
                         ▼
    ┌─────────────────────────────────────┐
    │          _Grunnable（可运行）         │ ← 等待被 M 挑中
    └─────────────────────────────────────┘
                         │  M 取到这个 G
                         ▼
    ┌─────────────────────────────────────┐
    │          _Grunning（正在运行）        │
    └──────┬──────────────┬───────────────┘
           │              │
    时间片用完/被抢占    主动阻塞(channel/sleep/锁)
           │              │
           ▼              ▼
      _Grunnable    _Gwaiting（等待中）
                         │  等待条件满足
                         ▼
                    _Grunnable
                         │
           ┌─────────────┼─────────────┐
           │ 执行完成      │              │
           ▼              ▼              ▼
      _Gdead（死亡）   _Gscan...    _Gsyscall
                     （GC扫描中）  （系统调用中）
```

**关键理解：**

- `_Gidle` → `_Grunnable`：`go func()` 创建 goroutine 后，它进入可运行状态，放在 P 的本地队列中
- `_Grunnable` → `_Grunning`：M 从队列中取出 G，开始执行
- `_Grunning` → `_Gwaiting`：G 执行了 `ch <-`、`time.Sleep`、`sync.Mutex.Lock` 等阻塞操作
- `_Grunning` → `_Grunnable`：G 的时间片用完了（10ms），被调度器放回队列，让其他 G 执行
- `_Gwaiting` → `_Grunnable`：阻塞条件满足（比如 channel 有数据了），G 被唤醒，重新进入可运行队列

### 3.2 M（Machine）——操作系统线程的代理

M 就是操作系统线程。但注意：**M 不是 Go 创建的，M 是 Go 向操作系统申请的**。Go 的调度器运行在 M 之上。

**M 里面存了什么？**

```go
// src/runtime/runtime2.go
type m struct {
    g0      *g     // 每个 M 都有一个特殊的 g0，用于执行调度逻辑
    curg    *g     // 当前正在执行的用户 goroutine
    
    p       puintptr // 当前绑定的 P（没有则为 nil）
    nextp   puintptr // 即将绑定的 P
    
    spinning bool   // 当前 M 是否在"旋转"——即没找到可用的 G，正在努力找
    
    // 系统调用相关
    locksg  *g      // 如果当前 M 在系统调用中被阻塞，这里记录它正在执行的 G
    
    // 链表
    schedlink muintptr // M 链表，用于全局 M 列表
    alllink   *m       // 所有 M 的链表
    
    // 等等...
}
```

**关键理解：g0 是什么？**

每个 M 都有一个特殊的 goroutine，叫 **g0**。g0 不执行用户代码，它只执行**调度逻辑**。当你写 `go func() { ... }()` 时，实际上是这样运作的：

```
1. 用户 G 在 M 上运行
2. 用户 G 调用 go func() → 创建新 G
3. M 切换到 g0 栈（调度栈）
4. g0 把新 G 放进 P 的本地队列
5. g0 调用 schedule() 函数选择下一个要运行的 G
6. M 从 g0 栈切换到下一个 G 的栈，继续执行用户代码
```

也就是说，**调度循环是在 g0 这个特殊 goroutine 上运行的**。这是一个非常巧妙的设计：调度器本身也是一个 goroutine（虽然很特殊），这样就统一了调度器和被调度对象的抽象。

### 3.3 P（Processor）——调度的核心中介

P 是 GMP 的**核心创新**。它代表执行 Go 代码所需的资源，主要职责是：

1. **为 M 提供运行环境**：M 必须绑定一个 P 才能执行 Go 代码
2. **持有本地 G 队列**：减少全局锁竞争
3. **管理调度相关的资源**：比如内存缓存（mcache）挂在 P 上，而不是 M 上

**P 里面存了什么？**

```go
// src/runtime/runtime2.go
type p struct {
    id          int32   // P 的 ID
    
    // 调度队列
    runqhead    uint32  // 本地可运行队列的头指针（环形队列）
    runqtail    uint32  // 本地可运行队列的尾指针
    runq        [256]guintptr // 本地可运行队列，最多 256 个 G（无锁访问！）
    runnext     guintptr // 下一个优先执行的 G（只有 1 个，特殊通道）
    
    // 内存管理
    mcache      *mcache // 每个 P 私有的内存缓存（分配小对象不需要全局锁！）
    
    // 状态
    status      uint32  // _Pidle / _Prunning / _Psyscall / _Pgcstop / _Pdead
    link        puintptr // 空闲 P 链表
    
    // 调度统计
    schedtick   uint32  // 调度次数（每次 schedule 后 +1）
    syscalltick uint32  // 系统调用次数
    
    // 等等...
}
```

**P 的本地队列为什么是 256？**

256 是精心选择的数字：
- 足够小：遍历成本低
- 足够大：大多数场景下 goroutine 不会超出这个范围，不需要去全局队列
- 环形队列：无锁访问（只有当前绑定的 M 操作它），头尾各有一个指针

**runnext 是什么？**

runnext 是一个"特殊通道"，只能放 1 个 G。设计意图是：

> 如果一个 G 刚创建出来，或者刚被唤醒，它很可能还需要 CPU 时间。把它放在 runnext 位置，下次调度时优先执行它，可以提高 CPU 缓存命中率。

本地队列的优先级顺序：**runnext > 本地 runq[256] > 全局队列 > 网络轮询器**

### 3.4 全局队列：最后的兜底

每个 P 的本地队列容量有限（256 + 1 个 runnext），当本地队列满了，新创建的 G 就会放入**全局队列**。此外，被抢占时间较长的 G 也可能被放入全局队列。

```go
// src/runtime/runtime2.go
type schedt struct {
    lock        mutex   // 保护全局队列的锁
    runq        gQueue  // 可运行的 G 队列
    runqsize    int32   // 全局队列中的 G 数量
    
    // 空闲的 M 链表（没有绑定 P 的 M）
    midle       muintptr
    
    // 空闲的 P 链表（没有被 M 绑定的 P）
    pidle       puintptr
    npidle      uint32  // 空闲 P 的数量
    nmspinning  uint32  // 处于 spinning 状态的 M 数量
    
    // 等等...
}
```

## 四、调度全流程：一个 goroutine 的一生

### 4.1 从 `go func()` 开始

当你写下这行代码：

```go
go func() {
    fmt.Println("hello")
}()
```

背后发生的事情：

```
步骤1：创建 G
   ┌──────────────────────────────────────┐
   │ runtime.newproc() → 从 P 的本地缓存 │
   │ 或全局堆分配内存，初始化 G 结构体     │
   │ G 状态设为 _Grunnable               │
   └──────────────┬───────────────────────┘
                  │
步骤2：放入队列
   ┌──────────────▼───────────────────────┐
   │ 尝试放入当前 P 的 runnext（优先）     │
   │ → runnext 有位置？直接放入 ✓         │
   │ → runnext 满了，尝试本地 runq[256]   │
   │ → runq[256] 满了，放入全局队列       │
   └──────────────┬───────────────────────┘
                  │
步骤3：调度
   ┌──────────────▼───────────────────────┐
   │ 当前 G 继续运行... 直到某时刻        │
   │ 调度器被触发 → schedule() → findrunnable() │
   │ → 从 runnext/本地队列/全局队列 取到新 G │
   │ → execute(g, ...)                   │
   └──────────────────────────────────────┘
```

### 4.2 调度器什么时候被触发？

调度不是随时发生的，它有明确的触发时机：

| 触发时机 | 说明 | 例子 |
|---------|------|------|
| **主动调度** | G 主动让出 CPU | `time.Sleep()`, `ch <-`, `sync.Mutex.Lock()` |
| **被动调度** | G 的时间片用完了（10ms） | sysmon 监控线程发现某个 G 运行太久，设置抢占标志 |
| **系统调用** | G 进入系统调用 | `read()`, `write()`, `network IO` |
| **GC 相关** | GC 需要 STW 或标记辅助 | GC 的标记阶段 |

### 4.3 schedule() 函数的核心逻辑

这是调度器最核心的函数。我用伪代码展示它的逻辑：

```go
// 伪代码：调度循环
func schedule() {
    _g_ := getg()  // 获取当前的 g0
    
    // 如果当前 M 没有绑定 P，或者 M 在自旋，那就去找一个 P
    if _g_.m.p == 0 {
        // 没有 P 就无法执行 Go 代码，必须获取一个
    }
    
top:
    // 第一步：检查是否有 GC 工作要做
    if GC 需要 STW {
        stopTheWorld()
    }
    
    // 第二步：每调度 61 次，强制去全局队列取一次 G
    // 这是为了防止全局队列中的 G 被"饿死"
    if _g_.m.p.ptr().schedtick % 61 == 0 {
        gp = 从全局队列取一个 G
        if gp != nil {
            goto execute
        }
    }
    
    // 第三步：从 runnext 取
    gp = _g_.m.p.ptr().runnext
    if gp != nil {
        goto execute
    }
    
    // 第四步：从本地队列取
    gp = runqget(_g_.m.p.ptr())
    if gp != nil {
        goto execute
    }
    
    // 第五步：从全局队列取
    gp = globrunqget(_g_.m.p.ptr(), 0)
    if gp != nil {
        goto execute
    }
    
    // 第六步：从网络轮询器取
    gp = netpoll(false)
    if gp != nil {
        goto execute
    }
    
    // 第七步（重点！）：工作窃取
    // 尝试 4 次，从其他 P 的本地队列偷 G
    for i := 0; i < 4; i++ {
        gp = 从随机选中的 P 偷取一半 G
        if gp != nil {
            goto execute
        }
    }
    
    // 第八步：实在找不到可执行的 G，再次检查网络轮询器和全局队列
    // 然后 M 进入休眠状态，等待被唤醒
    
execute:
    execute(gp)  // 切换到这个 G 并执行
}
```

**为什么是 61 这个数字？**

61 是一个质数。设计者想让访问全局队列的行为不那么规律——避免多个 P 在某个固定节奏一起访问全局队列，造成锁竞争。用质数可以打散访问时间点。

### 4.4 工作窃取（Work Stealing）——GMP 的精髓

当 P 的本地队列空了，它不是空闲着，而是会去**偷**其他 P 队列里的 G。这个过程叫 **Work Stealing**。

```
    P0 队列满了                  P1 队列空了
┌─────────────────┐         ┌─────────────────┐
│ [G1][G2][G3][G4]│         │ [  ]            │
│ [G5][G6][G7][G8]│         │                 │
└─────────────────┘         └────────┬────────┘
                                     │
                           P1 从 P0 偷取一半 G
                                     │
                                     ▼
                          ┌─────────────────┐
                          │ [G5][G6][G7][G8]│  偷了 4 个过来
                          └─────────────────┘
```

**偷取规则：**

1. 随机选一个 P（不是按顺序选，避免多个小偷同时盯上同一个 P）
2. 偷取该 P 本地队列的**后半部分**（runq 的后半段）。偷后半段的原因：前半段的 G 更可能被原 P 的 CPU 缓存"暖热"，后半段更久没有被执行
3. 最多尝试 4 次，4 次都没偷到就算了
4. 偷取过程中需要**短暂加锁**保护被偷的 P 的 runq（因为被偷的 P 可能也在操作自己的 runq）

**源码印证：**

```go
// src/runtime/proc.go  worksteal 简化版
func worksteal(_p_ *p, stealRunNextG bool) *g {
    // 随机选择起点
    stealTimersOrRunNextG := stealRunNextG
    
    // 遍历所有 P，尝试偷取
    for i := 0; i < 4; i++ {
        // 随机选一个 P
        p2 := allp[fastrand() % uint32(len(allp))]
        
        // 尝试从 p2 偷取
        if gp := runqsteal(_p_, p2, stealTimersOrRunNextG); gp != nil {
            return gp
        }
    }
    return nil
}
```

### 4.5 抢占式调度——不让一个 G 霸占 CPU

Go 1.14 之前，Go 的调度是**协作式**的——G 需要主动调用某些函数才会让出 CPU。这意味着如果一个 G 写了死循环：

```go
func main() {
    go func() {
        for {
            // 死循环，没有函数调用
        }
    }()
    time.Sleep(time.Second)
    fmt.Println("done")
}
```

在 Go 1.14 之前，这个程序永远不会打印 "done"，因为那个死循环的 goroutine 不会主动让出 CPU。

**Go 1.14 引入了基于信号的异步抢占，解决了这个问题。**

抢占的机制由 **sysmon**（系统监控线程）实现：

```
sysmon 线程（独立于 GMP，一直运行）
    │
    │ 每隔 10ms 检查一次
    ▼
检测到某个 G 运行超过 10ms
    │
    ▼
给该 G 所在的 M 发送 SIGURG 信号
    │
    ▼
M 收到信号，进入信号处理函数
    │
    ▼
信号处理函数修改 G 的栈，使其下次恢复时执行 schedule()
    │
    ▼
G 被换下，其他 G 得到执行机会
```

## 五、三大经典场景深度剖析

### 5.1 场景一：系统调用——G 阻塞了怎么办？

Goroutine 执行系统调用（如文件读写）时会阻塞。如果 M 傻等着，CPU 就浪费了。GMP 的处理方式是 **handoff（交接）**：

```
                        ┌───────────────────────────┐
                        │      sysmon 监控线程       │
                        └───────────────────────────┘
                                        │
                            检测到 M1 进入系统调用超过 20μs
                                        │
                                        ▼
     初始状态                     系统调用发生时
┌───────┐ ┌───────┐         ┌───────┐    ┌───────┐
│  P0   │ │  P1   │         │  P0   │    │  P1   │   P1 空闲
│ [G1]  │ │ [G3]  │         │ [G2]  │    │  ???  │   P1 的 runq 有 G3
│ [G2]  │ │       │         │       │    │       │
└───┬───┘ └───┬───┘         └───┬───┘    └───┬───┘
    │         │                  │             │
    ▼         ▼                  ▼             
┌───────┐ ┌───────┐         ┌───────┐    
│  M1   │ │  M2   │         │  M1   │    ① M1 带着 G1 进入系统调用（阻塞）
└───────┘ └───────┘         │  G1→  │    ② P0 和 M1 解绑
                             │ (阻塞) │    ③ P0 去寻找新的 M
                             └───────┘    
                                         结果状态
                              ┌───────┐    ┌───────┐
                              │  P0   │    │  P1   │
                              │ [G2]  │    │       │
                              └───┬───┘    └───┬───┘
                                  │             │
                                  ▼             ▼
                              ┌───────┐    ┌───────┐
                              │  M2   │    │  M3   │  ④ M3 是新创建的（或从空闲池取的）
                              │  G2→  │    │  G3→  │  ⑤ P0 绑定 M3，继续执行 G2
                              └───────┘    └───────┘
                                          ┌───────┐
                                          │  M1   │  M1 仍在系统调用中，等待返回
                                          │(阻塞中)│
                                          └───────┘
```

**关键步骤：**

1. M1 上的 G1 执行了系统调用 → M1 被操作系统挂起
2. sysmon 检测到 P0 的 M1 进入系统调用 → 将 P0 和 M1 解绑
3. P0 去空闲 M 列表找一个 M，如果没找到就新建一个 M2
4. P0 绑定 M2，M2 继续执行 P0 队列中的 G2
5. 当 M1 的系统调用返回后，G1 被重新标记为可运行。M1 尝试获取一个 P：
   - 如果有空闲 P，M1 绑定它，继续执行
   - 如果没有空闲 P，G1 被放入全局队列，M1 进入休眠

**核心思想：M 的数量可能多于 P 的数量，但正在执行 Go 代码的 M 数量 ≤ P 的数量。** 被阻塞在系统调用中的 M 不消耗 P。

### 5.2 场景二：协作式阻塞——channel、锁、sleep

当 goroutine 执行以下操作时，它不会阻塞 M：

```go
ch <- data          // channel 发送/接收
mu.Lock()           // 互斥锁
time.Sleep(1*time.Second) // 休眠
<-ctx.Done()        // context 取消
```

这些是**运行时管理的阻塞**，不是操作系统系统调用。处理流程完全不同：

```
M 正在执行 G1
    │
    ▼
G1 执行 ch <- data，但 channel 缓冲区满了
    │
    ▼
① runtime 把 G1 从 _Grunning 改为 _Gwaiting
② 把 G1 挂在 channel 的等待队列上（sendq 或 recvq）
    │
    ▼
③ M 调用 schedule() 选下一个 G
④ M 继续执行 G2
    │
    │ ... 一段时间后 ...
    │
    ▼
⑤ 另一个 G 从 channel 取走了数据，有缓冲区空间了
    │
    ▼
⑥ runtime 把 G1 从 _Gwaiting 改为 _Grunnable
⑦ G1 被放回 P 的本地队列
    │
    ▼
⑧ 下次调度时，G1 被 M 重新挑中，从 ch <- data 的下一行继续执行
```

**关键区别：**
- M **不会阻塞**，它立即换另一个 G 执行
- 整个切换过程都在**用户态**完成，不需要操作系统介入
- 这就是 goroutine 为什么这么轻量——阻塞只是修改几个字段，然后把控制权交还给调度器

### 5.3 场景三：网络 IO——netpoller 的魔法

网络 IO（socket 读写）是最高频的阻塞场景。Go 用 **netpoller（网络轮询器）** 优雅地解决了这个问题。

```
传统方式                     Go 的方式
┌──────────────┐           ┌─────────────────────────┐
│ 每个连接一个   │           │ 所有连接共用一个 netpoller │
│ 线程阻塞等待   │           │ goroutine 异步等待        │
│               │           │                          │
│ 10000 连接 =  │           │ 10000 连接 ≈              │
│ 10000 线程    │           │ 几个 goroutine           │
│ 内存爆炸！    │           │                          │
└──────────────┘           └─────────────────────────┘
```

**netpoller 工作流程：**

```
G1 执行 conn.Read()
    │
    ▼
① runtime 把 socket fd 注册到 netpoller（epoll/kqueue）
② G1 状态改为 _Gwaiting，挂在 netpoller 上
③ M 调用 schedule()，换另一个 G 执行
    │
    │ ... netpoller 在后台用 epoll 监听所有 fd ...
    │
    ▼
④ 数据到达！netpoller 收到事件通知
⑤ G1 状态改为 _Grunnable，放回 P 的本地队列
⑥ 下次调度时 G1 被 M 执行，Read() 返回数据
```

**netpoller 在调度循环中的位置：**

在 `findrunnable()` 函数中，当从本地队列和全局队列都找不到可运行的 G 时，调度器会调用 `netpoll(true)` 尝试从 netpoller 获取就绪的 G：

```go
// findrunnable() 中的逻辑
func findrunnable() (gp *g, inheritTime, tryWakeP bool) {
    // ... 前面的查找逻辑 ...
    
    // 实在找不到了，就去看看 netpoller 有没有就绪的
    if list := netpoll(true); !list.empty() {
        injectglist(&list)  // 把就绪的 G 注入运行队列
        // 再去找一次
    }
    
    // 还是找不到，M 进入休眠
    stopm()
    goto top
}
```

## 六、核心源码导读：带你读懂关键函数

Go 的调度器源码主要集中在 `src/runtime/proc.go`，这个文件大约 6000+ 行。你不用全看，只需要看这几个核心函数：

### 6.1 核心函数地图

```
入口：mstart()              ← 每个 M 的入口函数
  └── mstart1()
       └── schedule()       ← 调度循环的核心！选一个 G 执行
            │
            ├── findrunnable()  ← 从哪里找可运行的 G
            │    ├── runqget()       从本地队列取
            │    ├── globrunqget()   从全局队列取
            │    ├── netpoll()       从网络轮询器取
            │    └── worksteal()     从其他 P 偷取
            │
            └── execute()       ← 真正执行 G
                 └── gogo()     ← 汇编实现，切换到 G 的上下文

创建 goroutine：newproc()
  └── newproc1()
       └── runqput()       ← 把新 G 放入本地队列
            └── globrunqput()  ← 本地满了放全局

系统调用处理：
  reentersyscall()    ← 进入系统调用前
  exitsyscall()       ← 系统调用返回后
```

### 6.2 schedule() 源码节选

```go
// src/runtime/proc.go
// 调度器核心：找到一个可运行的 G 并执行它
func schedule() {
    mp := getg().m  // 获取当前 M
    
    // 安全检查：如果 M 持有锁，不能在持有锁时调度
    if mp.locks != 0 {
        throw("schedule: holding locks")
    }
    
top:
    pp := mp.p.ptr()
    pp.preempt = false
    
    // GC 相关的调度检查
    if gp := checkTimers(pp, 0); gp != nil {
        // ...
    }
    
    // 【关键】每 61 次调度，检查一次全局队列
    // 防止全局队列中的 G 饥饿
    if pp.schedtick%61 == 0 && sched.runqsize > 0 {
        lock(&sched.lock)
        gp := globrunqget(pp, 1)  // 从全局队列取 1 个
        unlock(&sched.lock)
        if gp != nil {
            execute(gp, inheritTime)
            // execute 不会返回，它会切换到 gp 的栈执行
        }
    }
    
    // 尝试从 runnext 取
    if gp := runnext; gp != nil && ... {
        execute(gp, inheritTime)
    }
    
    // 尝试从本地队列取
    if gp := runqget(pp); gp != nil {
        execute(gp, inheritTime)
    }
    
    // 尝试从全局队列取
    if gp := globrunqget(pp, 0); gp != nil {
        execute(gp, inheritTime)
    }
    
    // 尝试从 netpoller 取
    if netpollinited() && ... {
        if list := netpoll(false); !list.empty() {
            // 注入到运行队列，然后重新调度
            injectglist(&list)
            goto top
        }
    }
    
    // 都没找到，执行工作窃取
    if gp := runqsteal(pp, ...); gp != nil {
        execute(gp, inheritTime)
    }
    
    // 还是找不到...
    // M 进入休眠状态（stopm），等待被唤醒
    stopm()
    goto top
}
```

### 6.3 execute() 和 gogo()

```go
// execute 将 G 设置在 M 上执行
func execute(gp *g, inheritTime bool) {
    mp := getg().m  // 当前 M（g0）
    
    // 绑定 G 到 M
    mp.curg = gp
    gp.m = mp
    
    // 修改 G 的状态
    casgstatus(gp, _Grunnable, _Grunning)
    
    // 设置 G 的抢占标记
    gp.preempt = false
    gp.stackguard0 = gp.stack.lo + _StackGuard
    
    // 【关键】gogo 是汇编实现的
    // 它会把 M 从 g0 栈切换到 gp 的栈
    // 然后跳转到 gp.sched.pc（gp 上次被换下时的 PC 寄存器位置）
    gogo(&gp.sched)
}
```

`gogo()` 是一个汇编函数，做的事情非常底层：

1. 把 `gp.sched` 中保存的寄存器值恢复到 CPU 寄存器
2. 把栈指针切换到 gp 的栈
3. 跳转到 gp 上次离开时的指令地址（PC）
4. 此时 gp 从它的视角来看，就像"刚才的 schedule() 调用返回了"一样

**这就是协程切换的本质：保存当前上下文 → 恢复另一个上下文 → 跳转。** 整个过程在用户态完成，不需要操作系统介入。

### 6.4 系统调用交接源码

```go
// reentersyscall：准备进入系统调用
func reentersyscall(pc, sp uintptr) {
    gp := getg()
    
    // 把 M 上关键寄存器的值保存到 gp.sched
    save(pc, sp)
    
    // G 状态改为 _Gsyscall
    casgstatus(gp, _Grunning, _Gsyscall)
    
    // 把当前 P 标记为 _Psyscall
    pp := gp.m.p.ptr()
    pp.m = 0  // 临时解绑 P 和 M 的关系
    gp.m.p = 0
    
    // ... sysmon 会看到这个 P 处于 _Psyscall 状态
    // ... 如果超过阈值，sysmon 会把这个 P 拿走给其他 M
}

// exitsyscall：系统调用返回后
func exitsyscall() {
    gp := getg()
    
    // 尝试快速获取一个 P
    if oldp := gp.m.oldp; oldp != nil {
        // 如果原来的 P 还是空闲的，直接拿回来
        acquirep(oldp)
        casgstatus(gp, _Gsyscall, _Grunning)
        return
    }
    
    // 原来的 P 已经被拿走了
    // 尝试从空闲 P 列表中获取一个
    pp := pidleget()
    if pp != nil {
        acquirep(pp)
        casgstatus(gp, _Gsyscall, _Grunning)
        return
    }
    
    // 没有空闲 P 了！
    // G 进入全局队列等待
    globrunqput(gp)
    
    // M 进入休眠
    stopm()
    
    // 当 M 被唤醒时（因为某个 P 空闲了）
    // 重新获取 P，继续调度循环
    schedule()
}
```

## 七、常见面试题：手撕 GMP

### Q1：Go 为什么要用 GMP 而不是直接用线程？

**答：** 三个原因：

1. **线程创建成本高**：每个线程初始栈至少 1MB，goroutine 初始栈只有 2KB
2. **线程切换慢**：线程切换需要内核态介入，goroutine 切换完全在用户态，快 100 倍以上
3. **调度灵活性**：Go 的调度器知道 goroutine 之间的依赖关系（比如 channel），可以做更智能的调度

### Q2：P 的数量设置为多少比较合适？

**答：** 默认等于 CPU 核心数（`GOMAXPROCS`）。这是理论和实践都证明的最优值：

- 少了：CPU 核心没有被充分利用
- 多了：P 之间会产生无意义的调度竞争，增加上下文切换开销

**例外场景：**
- **IO 密集型应用**：可以适当调大（比如 CPU 核数 × 1.5），因为很多时间在等 IO，CPU 实际上是空闲的
- **CPU 密集型应用**：保持默认即可

### Q3：M 的数量有没有上限？

**答：** 有。Go 默认限制 M 的最大数量为 **10000**。你可以在源码中找到：

```go
// src/runtime/proc.go
// M 的最大数量
var maxmcount int32 = 10000
```

但实际上，M 的数量受 P 的数量约束——执行 Go 代码的 M 不会超过 P 的数量。超出的 M 都是陷入系统调用还没返回的。这个 10000 的限制主要是防止系统调用导致的 M 无限增长。

### Q4：如果一个 Goroutine 中有一个 `for {}` 死循环，其他 Goroutine 还能执行吗？

**答：** 在 Go 1.14 之后，**能**。因为引入了**基于信号的异步抢占**。

sysmon 线程会定期检测，如果发现某个 G 运行超过 10ms，就给它所在的 M 发 `SIGURG` 信号，强制该 G 让出 CPU。

但要注意：`for {}` 和 `for { i++ }` 之间有区别。`for {}` 里什么都没有，在某些场景下编译器可能会优化，抢占点的插入时机不同。但总体上，Go 1.14+ 已经解决了"死循环饿死其他 goroutine"的问题。

### Q5：工作窃取（Work Stealing）是什么？为什么偷后半部分？

**答：** Work Stealing 是 P 的本地队列为空时，从其他 P 的队列中"偷"任务的行为。

**偷后半部分的原因：**

1. 队列头部（前半部分）的 G 更可能被原 P 最近执行过，CPU 缓存还"热着"
2. 队列尾部（后半部分）的 G 等待时间更长，更"冷"，偷走它们对原 P 影响最小
3. 实现上，runq 是一个环形队列，偷后半部分只需要调整 tail 指针，不需要搬移数据

### Q6：GMP 模型中的全局队列和本地队列有什么区别？

| | 本地队列 | 全局队列 |
|---|---|---|
| **位置** | 每个 P 有一个 | 全局唯一 |
| **容量** | 256 + 1 (runnext) | 无限制 |
| **访问** | 无锁（只有绑定的 M 访问） | 需要加锁 |
| **用途** | 日常使用，绝大部分 G 在这里 | 当本地队列满了，或者需要均衡负载 |
| **优先级** | 高（优先从本地取） | 低（每 61 次调度才检查一次，或本地为空时） |

### Q7：G0 是什么？每个 M 都需要它吗？

**答：** G0 是一个特殊的 goroutine，**每个 M 都有一个**。它的特殊之处在于：

1. G0 的栈是**系统分配的固定栈**（约 8KB），不是 goroutine 的 2KB 初始栈
2. G0 **不执行任何用户代码**，只执行调度逻辑
3. 用户 G 和 G0 之间的切换发生在每次调度时

为什么需要 G0？因为调度器本身需要运行空间。如果直接在用户 G 的栈上运行调度代码，可能会破坏用户 G 的栈。G0 提供了一个"干净"的栈给调度器使用。

**记住三句话，GMP 就通了：**

1. **G 是任务，M 是干活的，P 是工作台**——M 必须拿到 P 才能执行 G
2. **P 的数量等于 CPU 核心数**——这是并发度的上限
3. **本地队列 + 工作窃取**——这是 GMP 高性能的核心秘密
