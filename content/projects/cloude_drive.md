# 前言
蓝山寒假的项目考核，本项目基于gin框架，构建的网盘服务，有用户功能与文件存储功能，采用领域设计，在domain层设计repo，Server接口。利用viper读取配置，实现缓存的三个防护。本这讲述项目介绍，关于使用ai的个人心得，学习路往分享。
# 一、📦 项目架构介绍
## 1. 🚪入口层（Entry）
 
- 核心职责：负责项目启动、初始化与生命周期管理。
- 核心模块：
-  cmd/main/main.go ：程序主入口，负责启动整个服务。
-  cmd/bootstrap/ ：启动引导模块，完成配置加载、路由注册、组件初始化等全局准备工作。
 
## 2. 🔌配置层（Config）
 
- 核心职责：统一管理所有配置信息，解耦配置与业务代码。
- 核心模块：
-  configs/ ：存放配置文件（如 YAML/JSON）与配置解析逻辑，支持多环境切换。
 
## 3. 🖥内部业务层（Internal）
 
这是业务逻辑的核心区域，严格遵循**领域驱动设计（DDD）**分层：
 
- API层：对外暴露 HTTP 接口，处理请求与响应。
-  routes/ ：定义路由规则，将请求分发到对应处理器。
-  handlers/ ：请求处理器，负责参数校验、调用服务层、返回标准化响应。
-  middlewares/ ：中间件，处理跨域、鉴权、日志、限流等横切关注点。
-  dtos/ ：数据传输对象，定义接口入参/出参结构，隔离内部模型与外部接口。
- 服务层（Service）：
-  service/ ：业务逻辑编排层，协调领域对象与数据访问层，实现核心业务流程。
- 数据访问层（Repo）：
-  repo/ ：数据持久化与查询层，封装数据库操作，为服务层提供数据能力。
- 领域层（Domain）：
-  domain/ ：领域核心，定义业务领域模型、领域规则与领域服务。
-  model/ ：数据模型层，映射数据库表结构，承载数据实体。
 
## 4. 🕹公共组件层（Pkg）
 
- 核心职责：提供可复用的基础能力，支撑各业务模块，避免重复造轮子。
- 核心模块：
-  auth/jwt ：JWT 身份认证与授权。
-  cache/ ：缓存防护封装。
-  db/ ：数据库连接池、ORM 封装与事务管理。
-  cron/ ：定时任务调度。
-  logger/ ：统一日志记录与日志级别管理。
-  save/ ：文件存储与上传下载能力。
-  conf/ ：配置读取与动态更新。
-  utils/ ：通用工具函数（如加密、字符串处理）。
-  errorer/ ：统一错误码与错误处理。
-  res/ redis连接封装。
 
## 5. 💾存储层（Storage）
 
- 核心职责：提供持久化存储能力。
- 核心组件：
-  MySQL ：关系型数据库，存储核心业务数据。
-  Redis ：内存数据库，用于缓存、会话与分布式锁。
-  文件系统 ：存储静态文件、上传资源等。

## 6、🔗 模块调用关系
 
- 启动流：MAIN  →  BOOT  → 加载  CFG 、初始化  ROUTES 、启动  CRON / LOGGER 。
- 请求流： ROUTES  →  MIDDLEWARES （鉴权/日志）→  HANDLERS （参数校验）→  SVC （业务逻辑）→  DOMAIN / REPO （数据处理）。
- 数据流： REPO  →  DB  →  MySQL ； SVC  →  CACHE  →  Redis ； SVC  →  SAVE  →  文件系统 。

# 二、🔑项目原则
1. 单个go程序包只能有一个主要函数方法（用于供外部调用，或实现接口），用小写函数折分防止过于臃肿
2. 单个go程序包命名必须简短且能体现真实现功能，用-区分模块
3. 尽量包装go内置函数，而不是直接使用，即使无额外设置
4. 尽量用var定义而非:=，犬其是非结构体
5. 尽量直接返回底层err

# 三、AI使用心得
## 1、使用了ai是否意味自身没有学习到经验
我的答案是否认的，使用ai不仅不会损失完成一个项目的经验，反倒可以让你更快速学习，更有经验地使用ai。为什么？一方面，如果对项目实现完全不清楚，是难以使用ai的，比如该用什么实现文件存储，普通人只有：实现文件存储这句话。但如果你明白的话，可以告诉他用什么工具，什么方式。其次，ai往往无法顾全整体，不清楚架构，很容易让你难以快速效验生成结果，甚至最后彻底将项目变成了明面上的"黑箱"箱，以至于以后只能变成你问我答的模式，随时间推移，bug越多，比如各种工具更新.......但你说，这些都可以让ai解决，那我问你，是你明白项目使用ai解决更好更快，还是啥都不懂的呢？人们会被一个工具取代，实在可笑，只知道恐慌，而不知道学习了，无论有没有AI都会被淘汰。
## 如何更好的使用ai？
这方面我只能表示自己的心得，像什么用mcp,skills,这方面更好的使用，目前没有实力分享。像什么用mcp调用各种工具，skills教导的AI该怎样更好的实现，我在做这个项目的时候并没有用到，也并不需要用到。有些时候确实很不喜欢这样专业化的名词，mcp就像定义了一个协议，让AI利用这个协议规则调用脚本，然后再返回用工具实现的数据，skills就像精细化的提示词，虽然这样解释可能并不完善且深入，但是其实更容易让人理解。就讲讲我的一些注意事项吧：
### 前期
- 利用AI学习知识，是最轻松的，把它理解为可以更智能搜索的浏览器，让他帮你总结知识，让你学习变得更加快速，不懂的点直接问，这是我目前感受到AI最舒服最好的地方。
- 利用AI，搭建初始架构，一个空的项目往往很难开始，利用AI先搭建一些基础架构，然后再自己慢慢修改
- 利用AI逐讲解每个语法，开始项目后，哪怕学到了知识，忘了一些，通过实践操作，让AI帮你复习，反倒可以记得更深更牢，你可能觉得完成这些功能，并不是自己的实力，但其实你只要看得懂，明白怎么实现的就行。你要成为的是一个工程师，而不是码农。一个农民割麦子，用机器去割，难不成就不是他自己割的？
### 中期
- 这时往往比较困难，你可能会遇到各种报错，那种AI解决反复失败，实际上就是你可能犯一些很简单的错误。很可能是某些配置写错（不给详细的提示词，ai很容易搞错一些配置，尤其是自己还参与在里面修改），这个时候你得有耐心，学会自己去读，我者尝试用不同的ai互相检查。尤其是要详细查看报的err。
- 这个时候尽量减少对AI的依赖，不要超前设计，一步一步来，完成很多细小简单的功能，添加需要修改大部分代码的功能
- 尝试用AI帮你写文档，一步一步执行，如trae的plan模式
### 后期
- 形我自己的代码风格后，要提示让ai按照你当前的架构进行修改，实际上，这个时候手动去修改反倒比AI迅速，比如我handler层直接调用的repo层，让ai修改，很容易造成后续难以维护，自己修改，其实也就命个名，然后疯狂按tap 的事。
- 学会自己总结提示词
### 注意
- 目前我看来的是，如果你想大幅度依靠AI，那就完全依靠，而不是让自己手动参与修改。
- 以上仅仅是我个人心得，还有很多不足的地方，仅供参考
# 学习分享
遇到的最大困难大概就是很多概念不清，先尝试自己去使用工具，千万不要困在解决某个bug上，否则会浪费很多时间，遇到bug让AI分析，解决不了，就搁在这里，尝试其他路径。学会看别人的项目，看网上的视频，不要埋头苦干，这会消耗你的心力。然后就是要脚踏实地，不要轻言放弃，为什么要说这两句大话？因为你在学习编程的路上，很容易接触到各种大佬，很容易被打击，很容易放弃。我能给你的鼓励就是，不要害怕技术不行，尝试做自己的项目，可能在模仿他们的路上，你根本追不上，但是你可以走自己的路。一定要尝试在github上学习，哪怕你根本看不懂，你要多停留在那个页面上，像刷短视频一样刷项目。
